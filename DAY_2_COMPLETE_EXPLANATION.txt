FILE 1: app/config.py
═══════════════════════════════════════════════════════════════════════════════

PURPOSE:
Read your .env file once at startup and make all settings available as a Python object
throughout your app. Instead of os.environ.get("KEY") scattered everywhere, you have
one clean Settings object.

WHY THIS IS BETTER:
- Type hints: IDE autocomplete works
- Validation: Pydantic checks types on startup, not at runtime
- One place to change config: update .env, everything updates
- Clear what's required: missing required var = error on startup (not a mysterious crash later)


CODE EXPLANATION:

```python
from pydantic_settings import BaseSettings, SettingsConfigDict
from functools import lru_cache

class Settings(BaseSettings):
```
→ Inherit from BaseSettings (not BaseModel). BaseSettings knows to read .env files.

```python
    APP_NAME: str = "DocuMind"
    DEBUG: bool = False
```
→ Type hints with default values. If .env has APP_NAME, it overwrites the default.
→ If .env doesn't have it, the default is used.

```python
    DATABASE_URL: str
```
→ NO default value = REQUIRED. If .env doesn't have DATABASE_URL, Pydantic raises an error
  on startup before the app even boots. You catch config errors immediately.

```python
    model_config = SettingsConfigDict(
        env_file=".env",
        extra="ignore",
    )
```
→ env_file=".env": read from this file
→ extra="ignore": if .env has RANDOM_VAR=value that we don't use, ignore it (no error)

```python
@lru_cache()
def get_settings() -> Settings:
    return Settings()
```
→ @lru_cache() is a Python decorator that caches the function result
→ First call: creates Settings object by reading .env
→ Future calls: returns the cached object from memory
→ Why: we don't want to re-read .env on every request (slow + wasteful)
→ settings = get_settings() at the module level means we read .env once on app startup

FILE 2: app/database.py
═══════════════════════════════════════════════════════════════════════════════

PURPOSE:
Set up SQLAlchemy's engine (connection to Postgres), session factory (connection pool),
and the Base class that all your models inherit from.

WHY THIS FILE:
- Centralizes database setup (one file, one place to debug)
- Provides a dependency injection function (get_db) for FastAPI
- Defines Base for all models to inherit from


CODE EXPLANATION:

```python
from sqlalchemy import create_engine
from sqlalchemy.ext.declarative import declarative_base
from sqlalchemy.orm import sessionmaker, Session

from app.config import settings
```
→ Import everything we need. settings comes from config.py (reads .env)

```python
engine = create_engine(
    settings.DATABASE_URL,
    pool_pre_ping=True,
    pool_size=5,
    max_overflow=10,
)
```
→ create_engine() opens a connection to Postgres
→ settings.DATABASE_URL is like "postgresql://postgres:postgres@db:5432/documind"
→ pool_pre_ping=True: before using a connection, test it with SELECT 1
  (prevents "connection closed" errors if Postgres restarts while app is running)
→ pool_size=5: keep 5 connections open simultaneously
→ max_overflow=10: allow up to 10 extra connections when demand spikes (connection pool doesn't exceed 15)

```python
SessionLocal = sessionmaker(
    autocommit=False,
    autoflush=False,
    bind=engine,
)
```
→ SessionLocal is a FACTORY — call it to make new session objects
→ autocommit=False: we manually call session.commit() — nothing saves until we say so
→ autoflush=False: we control when SQLAlchemy sends SQL to the DB
→ bind=engine: use the engine we just created

```python
Base = declarative_base()
```
→ Base is the parent class for all your models
→ When you do class User(Base): SQLAlchemy knows this is a table definition
→ Base.metadata.create_all() later will create all tables defined by classes inheriting from Base

```python
def get_db() -> Generator[Session, None, None]:
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```
→ This is a FastAPI DEPENDENCY — a function that runs before each request handler
→ SessionLocal() creates a NEW session for this request
→ yield db: give the session to the route handler
→ finally: db.close() ALWAYS runs (even if the route handler crashes)
→ Why finally: prevents connection leaks (sessions stuck open)
→ How to use: def my_route(db: Session = Depends(get_db)):

→ Generator[Session, None, None] is a type hint that says:
  "this function yields a Session object, receives nothing from the caller, returns nothing"

```python
def create_tables():
    Base.metadata.create_all(bind=engine)
```
→ Called at app startup (from main.py)
→ Looks at all classes that inherit from Base (User, Document, QueryLog etc.)
→ For each one, CREATE TABLE IF NOT EXISTS in Postgres
→ Won't overwrite existing data — safe to call multiple times

FILE 3: app/models/user.py
═══════════════════════════════════════════════════════════════════════════════

PURPOSE:
Define what the "users" table looks like in Postgres.
This is SQLAlchemy ORM code — it maps Python classes to database tables.

KEY CONCEPT: Model vs Schema
- Model (this file): talks to DATABASE — SQLAlchemy columns map to Postgres columns
- Schema (auth.py):  talks to HTTP — Pydantic validates request/response JSON
They're separate because API input ≠ DB storage
  (API gets "password", DB stores "hashed_password")


CODE EXPLANATION:

```python
from sqlalchemy import Column, Integer, String, Boolean, DateTime
from sqlalchemy.sql import func
from app.database import Base

class User(Base):
    __tablename__ = "users"
```
→ Inherit from Base (from database.py)
→ __tablename__ = "users" tells SQLAlchemy to create a table called "users" in Postgres

```python
    id = Column(
        Integer,
        primary_key=True,
        index=True,
        autoincrement=True,
    )
```
→ id is the PRIMARY KEY (unique identifier for each row)
→ Integer type → SQL INT
→ primary_key=True: Postgres enforces uniqueness + makes this the main identifier
→ index=True: Postgres creates an index (faster lookups by id)
→ autoincrement=True: Postgres auto-assigns 1, 2, 3... when rows are inserted

→ In Postgres this becomes: id INT PRIMARY KEY AUTO_INCREMENT

```python
    email = Column(
        String,
        unique=True,
        index=True,
        nullable=False,
    )
```
→ String type → SQL VARCHAR
→ unique=True: Postgres constraint — no two rows can have the same email
→ index=True: speeds up queries like "find user by email"
→ nullable=False: SQL NOT NULL constraint — email is required

→ In Postgres: email VARCHAR NOT NULL UNIQUE

```python
    hashed_password = Column(
        String,
        nullable=False,
    )
```
→ NEVER store plain text passwords
→ We store bcrypt hashes (60 characters) in this column
→ When user logs in, we hash their submission and compare hashes

→ In Postgres: hashed_password VARCHAR NOT NULL

```python
    is_active = Column(
        Boolean,
        default=True,
    )
```
→ Boolean type → SQL BOOLEAN
→ default=True: when we create a new User without specifying is_active,
  SQLAlchemy sets it to True

→ In Postgres: is_active BOOLEAN DEFAULT TRUE

```python
    created_at = Column(
        DateTime(timezone=True),
        server_default=func.now(),
    )
```
→ DateTime(timezone=True) → SQL TIMESTAMP WITH TIME ZONE (stores timezone info)
→ server_default=func.now(): the DATABASE (Postgres) sets this to NOW() when row is inserted
→ We DON'T pass created_at from Python — the DB does it automatically
→ Why: ensures all timestamps are consistent (DB server time, not client time)

→ In Postgres: created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()

```python
    updated_at = Column(
        DateTime(timezone=True),
        server_default=func.now(),
        onupdate=func.now(),
    )
```
→ server_default=func.now(): set to NOW() on INSERT
→ onupdate=func.now(): Postgres updates this to NOW() on every UPDATE
→ So if we call db.commit() after changing a user, updated_at automatically updates

→ In Postgres: updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()

```python
    def __repr__(self):
        return f"<User id={self.id} email={self.email}>"
```
→ Python magic method — controls what print(user) shows
→ Without this, you'd see <__main__.User object at 0x...> (not helpful)
→ With this, you see <User id=1 email=alice@example.com> (much clearer for debugging)

PURPOSE:
Define the shapes of data coming IN (request bodies) and going OUT (responses) of your API.
This is Pydantic code — it validates HTTP JSON, not database data.

KEY: Schemas validate HTTP. Models are database tables. They look similar but are separate.


CODE EXPLANATION:

```python
from pydantic import BaseModel, EmailStr, Field
from datetime import datetime

class UserRegister(BaseModel):
    email: EmailStr
    password: str = Field(min_length=8, description="Minimum 8 characters")
```
→ BaseModel: Pydantic class for HTTP request/response validation
→ This schema defines the body of POST /auth/register

→ email: EmailStr means:
  - FastAPI AUTOMATICALLY validates it's a real email format
  - "notanemail" → automatic 422 Unprocessable Entity error
  - You don't write the validation code — Pydantic does it

→ password: str = Field(min_length=8)
  - FastAPI AUTOMATICALLY validates length >= 8
  - "pass" → 422 error automatically
  - Field() adds metadata (min_length, description for docs)

→ When FastAPI receives POST /auth/register with {"email": "x@y.com", "password": "pass123"}
  it creates a UserRegister object and passes it to your route handler
→ If validation fails, FastAPI returns 422 WITHOUT running your code

→ This saves you from writing:
  if not isinstance(email, str): raise...
  if len(password) < 8: raise...
  ... for every route. Pydantic does it.

```python
class UserResponse(BaseModel):
    id: int
    email: str
    is_active: bool
    created_at: datetime

    model_config = {"from_attributes": True}
```
→ Defines what we RETURN after successful register

→ Notice: NO hashed_password, NO updated_at
→ We only return what the client needs
→ Sensitive fields (hashed_password) are left out

→ model_config = {"from_attributes": True}:
  - Tells Pydantic to read from SQLAlchemy model attributes
  - Without this, Pydantic only reads dicts
  - So when we return a SQLAlchemy User object, Pydantic can extract id, email etc. from it

→ When you do return new_user (a SQLAlchemy User object):
  FastAPI serializes it through UserResponse, returning only those 4 fields as JSON

```python
class TokenResponse(BaseModel):
    access_token: str
    token_type: str = "bearer"
```
→ Returned after POST /auth/login
→ access_token is the JWT string
→ token_type = "bearer" is a default — most responses have this

```python
class TokenData(BaseModel):
    email: str | None = None
```
→ This is what we DECODE from a JWT token
→ "sub" (subject) claim in the token contains the email
→ We use this in dependencies.py to verify the token

PURPOSE:
Pure business logic for password hashing and JWT creation.
NO HTTP here, NO database here — just pure Python functions.
This allows you to test auth logic without running a web server.


CODE EXPLANATION:

```python
from passlib.context import CryptContext

pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")
```
→ CryptContext manages password hashing
→ schemes=["bcrypt"]: use bcrypt as the hashing algorithm
  - bcrypt is intentionally slow (prevents brute-force attacks)
  - includes a random salt (same password → different hash each time)
→ deprecated="auto": if we switch algorithms later, passlib auto-rehashes on next login

```python
def hash_password(plain_password: str) -> str:
    return pwd_context.hash(plain_password)
```
→ Takes "mypassword123" and returns "$2b$12$K8Nv3q5Xm..." (60-char hash)
→ The hash includes the salt inside it
→ Same password hashed twice = different hashes (random salt)
→ This is CORRECT and expected behavior

→ NEVER do: if password == stored_password
→ ALWAYS do: if hash_password(submitted) == stored_hash? NO, use verify_password()

```python
def verify_password(plain_password: str, hashed_password: str) -> bool:
    return pwd_context.verify(plain_password, hashed_password)
```
→ Takes the plain text password and the stored hash
→ bcrypt extracts the salt from the hash
→ Hashes the plain password with that salt
→ Compares the result to the stored hash
→ Returns True if they match

→ Why this works:
  hash("password123") with salt1 = "$2b$12$K8N..."
  hash("password123") with salt1 = "$2b$12$K8N..."  ← same result
  hash("wrongpassword") with salt1 = "$2b$12$XyZ..."  ← different result

```python
from datetime import datetime, timedelta, timezone
from jose import jwt

def create_access_token(
    data: dict,
    expires_delta: Optional[timedelta] = None
) -> str:
```
→ Takes a dict (typically {"sub": "user@email.com"}) and creates a signed JWT
→ returns a string like "eyJ0eXAiOiJKV1QiLCJhbGci..."

```python
    to_encode = data.copy()
```
→ Copy the dict so we don't mutate the original
→ Python dict operations are passed by reference — data would be modified if we didn't copy

```python
    if expires_delta:
        expire = datetime.now(timezone.utc) + expires_delta
    else:
        expire = datetime.now(timezone.utc) + timedelta(
            minutes=settings.ACCESS_TOKEN_EXPIRE_MINUTES
        )
```
→ Calculate when the token expires
→ timezone.utc: always use UTC, never local time (prevents bugs across timezones)
→ If expires_delta is passed, use it; otherwise use the setting (30 minutes)

```python
    to_encode.update({"exp": expire})
```
→ Add "exp" (expiry) claim to the token data
→ "exp" is a standard JWT claim — jose automatically checks it on decode

```python
    encoded_jwt = jwt.encode(
        to_encode,
        settings.SECRET_KEY,
        algorithm=settings.ALGORITHM,
    )
    return encoded_jwt
```
→ jwt.encode() creates the JWT:
  1. Encodes the payload as base64
  2. Signs it with SECRET_KEY using HS256
  3. Returns "xxxxx.yyyyy.zzzzz"

→ The structure is:
  - xxxxx: base64({"alg": "HS256", "typ": "JWT"})
  - yyyyy: base64({"sub": "user@email.com", "exp": ...})
  - zzzzz: HMACSHA256(xxxxx.yyyyy, SECRET_KEY)

→ If anyone tampers with the payload, the signature won't match
→ If they don't have SECRET_KEY, they can't create a valid signature

PURPOSE:
Main FastAPI application. Sets up the app, middleware, error handlers, and includes routers.

KEY CONCEPTS:
- Lifespan: startup/shutdown code
- Routers: group related endpoints
- Middleware: runs before every request (CORS, logging etc.)
- Exception handlers: catch errors globally


CODE EXPLANATION:

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI, Request
from fastapi.middleware.cors import CORSMiddleware
from fastapi.responses import JSONResponse

from app.database import create_tables
from app.api.routes import auth

@asynccontextmanager
async def lifespan(app: FastAPI):
    # ── STARTUP ───────────────────────────────────────────────────────────
    print("🚀 DocuMind starting up...")
    create_tables()
    print("✅ Database tables ready")
    
    yield  # app runs here
    
    # ── SHUTDOWN ──────────────────────────────────────────────────────────
    print("🛑 DocuMind shutting down...")
```
→ @asynccontextmanager: run code at startup AND shutdown
→ Everything before yield: startup code
→ yield: hands control to FastAPI, app runs
→ Everything after yield: shutdown code

→ create_tables(): SQLAlchemy creates users, documents, query_log tables in Postgres
  (if they don't exist)
→ Runs once on app startup

```python
app = FastAPI(
    title="DocuMind API",
    description="...",
    version="0.1.0",
    docs_url="/docs",
    redoc_url="/redoc",
    lifespan=lifespan,
)
```

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```
→ CORS (Cross-Origin Resource Sharing): allows browsers to call your API
→ allow_origins=["*"]: allow any domain (fine for dev, lock down in production)
→ allow_methods=["*"]: allow GET, POST, PUT, DELETE etc.
→ allow_headers=["*"]: allow any headers (including Authorization)

→ Without this, browser requests fail with CORS error

```python
@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    return JSONResponse(
        status_code=500,
        content={
            "detail": "Internal server error",
            "type": type(exc).__name__,
        },
    )
```
→ Catches ANY unhandled exception in the entire app
→ Returns clean JSON instead of a raw Python traceback
→ Status 500: generic server error

→ Without this, if your code crashes, the client sees the full Python stack trace
→ With this, you hide internals but still know the error type

FLOW SUMMARY: What happens when you POST /auth/register
═══════════════════════════════════════════════════════════════════════════════

1. Client sends: POST /auth/register with {"email": "alice@x.com", "password": "pass123"}

2. FastAPI reads the JSON body and VALIDATES it against UserRegister schema
   - Checks email is valid format → if not, return 422
   - Checks password length >= 8 → if not, return 422

3. FastAPI calls get_db() dependency
   - SessionLocal() creates a DB session
   - Yields it to the route handler

4. register() function runs with user_data and db
   - Checks if email exists in DB
   - If yes → raise 400
   - Calls hash_password() to bcrypt the password
   - Creates User() ORM object
   - db.add() + db.commit() inserts to Postgres
   - db.refresh() gets the auto-generated id and created_at

5. register() returns new_user (a SQLAlchemy User object)

6. FastAPI SERIALIZES the response through UserResponse schema
   - Only includes: id, email, is_active, created_at
   - Strips out: hashed_password
   - Converts to JSON

7. FastAPI returns HTTP 201 with JSON body

8. After route completes, finally: db.close() closes the DB session


═══════════════════════════════════════════════════════════════════════════════
FLOW SUMMARY: What happens when you GET /auth/me with a token
═══════════════════════════════════════════════════════════════════════════════

1. Client sends: GET /auth/me
   Headers: Authorization: Bearer eyJ0eXAi...

2. FastAPI extracts the token from the Authorization header (oauth2_scheme does this)

3. FastAPI calls get_current_user() dependency
   - Calls jwt.decode(token, SECRET_KEY, algorithms=["HS256"])
     → verifies signature
     → checks expiry
     → returns {"sub": "alice@x.com"}
   - Extracts email from payload
   - Queries DB for user by email
   - Returns User object if found and is_active=True
   - Otherwise raises 401 (token invalid/expired, user not found, user disabled)

4. get_me() function runs with current_user (the User object from above)
   - If we got here, the token is valid and the user exists

5. Return current_user

6. FastAPI SERIALIZES through UserResponse schema

7. FastAPI returns HTTP 200 with user JSON


These flows happen EVERY request. The dependency system handles auth automatically.
